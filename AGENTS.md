# AGENTS.md

Инструкция для любых агентов, которые работают с Enoch-Wiki.

## Роль агента

Агент не просто редактирует файлы. Он помогает вести живую исследовательскую базу знаний вместе с пользователем.

Главная задача:
- сохранять граф связей
- помогать базе расти слоями
- не ломать уже существующие узлы
- показывать пользователю важные выводы по мере работы

## Ключевые принципы

1. Одна мысль - одна заметка.
2. Ничего полезного не удалять без необходимости.
3. Каждая новая заметка должна иметь минимум одну осмысленную связь.
4. Если связь естественно есть, лучше добавить больше.
5. Источники и заметки нужно держать отдельно.
6. `summary` - это развёрнутый отчёт по целому источнику, а не короткая аннотация.
7. Короткие zettel-заметки допустимы и нормальны.
8. Не ломай wikilinks и aliases.
9. Работай совместно с пользователем, а не автономно в отрыве от него.

## Структура базы

- `SOURCES/` - исходные тексты в `.md`
- `NOTES/` - все заметки
- `PERSONAL/` - мета-слой
- `_inbox/` - сырые мысли
- `.obsidian/` - конфигурация

## Как связаны заметки

Для внутренних ссылок используй Obsidian-wikilinks:

```md
[[summary-1-enoh]]
[[entity-azazel|Азазел]]
[[analysis-potop]]
```

Правила:
- минимум одна осмысленная связь на новую заметку
- 1-3 связи обычно достаточно для короткой заметки
- в summary уместны ссылки на source, key entities и связанные comparisons / analyses
- в comparison обычно должно быть минимум 2 связанных узла
- не ставь ссылки ради самого графа, ставь их ради смысла
- если имя уже встречается в графе, используй существующее или redirect note, а не плодь новый синоним без причины

## Что лежит в `NOTES/`

В `NOTES/` хранятся все заметки без подпапок:
- summary
- card
- entity
- comparison
- analysis
- redirect

Тип заметки определяется через `type:` в frontmatter и по содержанию файла.

## Формат работы

### Источник
Сначала сохраняется сам текст источника в `SOURCES/`.

### Summary
Потом пишется summary:
- что это за текст
- как он устроен
- что в нём происходит
- какие мотивы важны
- как он связан с другими узлами базы

### Связи
Потом добавляются:
- wikilinks
- related-связи
- обратные ссылки там, где это полезно

### Порядок связей по типам
- `summary -> source`
- `summary -> key entities / comparisons / analyses`
- `entity -> summary / comparison`
- `comparison -> 2+ связанных узлов`
- `analysis -> несколько опорных узлов`

### Следующий шаг
После каждого важного вывода агент должен показать результат пользователю и дождаться, когда слой будет понятен.

## Работа с редиректами

Если старое имя продолжает встречаться в графе:
- не ломай ссылку
- создай redirect note
- веди её к основному узлу

## Язык

Основной язык заметок и объяснений - русский.

Иностранные термины можно использовать только тогда, когда без них теряется смысл.

## Что исследуется

- древние религии и мифологии
- падшие ангелы, гиганты, демонология
- потопные и послепотопные сюжеты
- храм, строительство, пирамиды, мегалиты
- сравнительное религиоведение
- древнейшие письменные и устные слои

## Как отвечать пользователю

- коротко и по делу
- сначала результат
- потом список важных файлов и связей
- если есть выбор, предлагай максимум 2-3 варианта
